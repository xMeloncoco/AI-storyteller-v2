# Working Style

## Change Analysis Protocol

When a suggested or requested change has meaningful impact on behavior, architecture, or logic (not minor visual tweaks), always:

1. Analyze what it does and why it works or doesn't
2. List pros and cons
3. Give a clear recommendation — including how to do it differently or better if relevant
4. Only flag alternatives when they genuinely improve things, not for the sake of it
5. Implement once aligned

This applies to any change significant enough to affect how the app works, is structured, or behaves over time. Skip the analysis step for minor visual/cosmetic changes.

Always take the build instructions file into account when analyzing or implementing changes.
